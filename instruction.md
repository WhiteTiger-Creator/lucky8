Plan the Sentinel-1 remediation containment for a set of proposed remediation bundles and report the contained severity.

During incident response a set of candidate remediation bundles is proposed; each bundle has a severity weight and locks a set of shared assets, and two bundles conflict when they lock a common asset. The plan decides how much severity can be contained by applying a compatible set of bundles at once, records the selected set and its derived aggregates, and emits integrity checksums.

How the plan is computed is **not** in the output contract and you must reconstruct it. Bundle canonicalization and its validity bounds, the containment objective itself, the deterministic tie-broken selected set, and the residual packing were settled by the Sentinel-1 incident-response board, and `/app/incident/remediation_review_log.md` is the authoritative record of every one of them. Most of the log is routine noise, and the February draft proposals were partly reversed later; several were revised again in the 2026-05 review, so trace each rule to its final decision — where a draft or interim conflicts with a later decision, the later decision governs. The natural first guesses at the objective are decoys the board explicitly considered and rejected; read the log rather than assume.

`/app/docs/plan_spec.md` is the output contract only: it fixes the input shape, the exact `plan.json` key set, and the byte-level checksum serialization — not how the values are derived.

Given the bundles in `/app/data/remediation.json`, compute the plan and write `plan.json` with the keys the spec lists. Implement the computation as `/app/remediate.py`, reading `--input` (default `/app/data/remediation.json`) and writing under `--output-dir` (default `/app/output`), and run it so `/app/output/plan.json` is in place. Derive every value from the input and the governing decisions; don't read or import anything from `/tests` or `/solution`.
