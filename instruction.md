Plan the Sentinel-1 incident containment for a set of proposed remediation bundles and report the contained threat severity.

A confirmed intrusion has left a set of assets compromised. Incident responders have proposed candidate remediation bundles; each bundle carries a threat-severity weight and must quarantine a set of assets to be applied, and two bundles conflict when they quarantine a common asset, because an asset cannot be torn down twice in the same containment pass. The plan determines how much threat severity can be contained by one compatible set of bundles, records the selected set and the exposure that remains uncontained, assigns the surviving bundles to responder tiers, and emits integrity checksums for the containment record.

The evaluation rules are not part of the output contract. `/app/incident/remediation_review_log.md` is the authoritative source for all of them: bundle canonicalization and its validity bounds, the containment objective, the deterministic tie-broken contained set, the residual packing, the severity-tier and asset-pressure scoring, the response-urgency ledger, and the response-wave load, admission and tier rules.

The log records these rules as dated review entries, and more than one entry addresses each rule: February draft proposals and April interim decisions were revised by the 2026-05 incident review. For every rule the governing value is the one stated in its latest dated decision. Earlier drafts and interim entries are superseded and carry no force, including where a superseded entry states an otherwise plausible alternative.

`/app/docs/plan_spec.md` is the output contract only: it fixes the input shape, the exact `plan.json` key set, and the byte-level checksum serialization — not how the values are derived.

Two output-vocabulary and type facts the verifier matches literally: a response-wave bundle's tier is exactly one of the lowercase strings `immediate`, `urgent`, or `routine`, and `response_tier_counts` always enumerates all three in that order, emitting `0` for a tier with no bundles; `response_order` is a JSON array in the log's response ordering, which is not ascending id order — `response_wave_ids` is the sorted one.

The remediation policies are read from `/app/data/remediation_policies.json` at that fixed absolute path; `--input` selects the bundle file only and never relocates the policy file. The wave and urgency thresholds are not global constants: they are resolved per bundle from that file by the bundle's severity band, through a layered default-then-override inheritance whose tiers, banding and serialization the review log specifies.

Given the bundles in `/app/data/remediation.json`, produce the containment plan at `/app/output/plan.json` with the keys the spec lists, via `/app/remediate.py` reading `--input` (default `/app/data/remediation.json`) and writing under `--output-dir` (default `/app/output`). Every value derives from the input and the governing decisions; nothing is read or imported from `/tests` or `/solution`.
