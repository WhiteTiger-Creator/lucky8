Plan the Sentinel-1 remediation containment for a set of proposed remediation bundles and report the contained severity.

During incident response a set of candidate remediation bundles is proposed; each bundle has a severity weight and locks a set of shared assets, and two bundles conflict when they lock a common asset. The key quantity is how much severity can be contained by applying a set of **asset-disjoint** bundles at once — bundles that compete for a shared asset cannot all be applied, so the contained severity is a maximum-weight asset-disjoint packing, deliberately **not** the sum of all bundle severities (that over-counts shared assets) and **not** a greedy highest-severity-first selection (greedy is not optimal for this packing). `/app/docs/plan_spec.md` defines the input format, the canonicalization rules, the packing objective, and the exact output schema and checksum serialization. Read it closely — the objective is an exact packing, and both the total-severity sum and the greedy selection are wrong answers.

Given the bundles in `/app/data/remediation.json`, compute the plan and record the results in `plan.json`: the bundle count, the total proposed severity, the largest single bundle severity, the exact `max_contained_severity`, and the `bundle_checksum` and `plan_checksum` over the exact serializations the spec defines.

Implement the computation as `/app/remediate.py`, reading `--input` (default `/app/data/remediation.json`) and writing under `--output-dir` (default `/app/output`), and run it so `/app/output/plan.json` is in place. Derive every value from the input; don't read or import anything from `/tests` or `/solution`.
