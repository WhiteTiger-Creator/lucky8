# Tideguard network-flow intrusion containment

Act as the security operations engineer responding to the Tideguard intrusion. The tampered flow-quarantine automation left persistence and escalation paths on the collector host: a rogue automation key in root's `authorized_keys`, a passwordless sudoers drop-in, and a world-readable flow-collector credential. **First contain the host** to match `/app/docs/containment_runbook.md` — revoke the automation's SSH persistence without locking out the on-call bastion key, remove its privilege escalation entirely, and lock down the exposed credential.

Then restore the quarantine control at `/app/workflow/export_report.py`. It reads the raw flow-telemetry stream off the sensors and decides which hostile flows on each subnet are isolated from the fleet and escalated to responders; while it is tampered it releases malicious flows back onto the network instead of containing them. It always reads the isolation and inspection control windows from their fixed absolute path `/app/data/flow_policies.json`: `--input` reparameterizes only the telemetry stream, so the policies file never becomes relative to the input or output directory and is read from that same absolute path even when the telemetry is loaded from elsewhere. It must again leave three records responders act on: the incident summary `summary.json`, the per-subnet exposure map `subnet_matrix.json`, and the quarantine queue `quarantined.jsonl`.

No triage tooling survived the tampering, so you must **create** `/app/flow_audit.py` — it does not exist yet — exposing two actions. `diagnose --dossier PATH --report PATH` reads the review log at the `--dossier` path and **writes** its report of the six known containment defects, with the forensic evidence for each, to the `--report` path, which is an output file it creates (not an input to read). `repair --output-dir PATH` restores the control from its frozen pre-incident snapshot, re-runs containment, and records the containment diagnosis (`diagnosis.json`) and the repair audit (`repair_audit.json`) beside the three records, defaulting to `/app/output`. The repair audit's `forbidden_tokens_removed` lists exactly the spec's `workflow_repair.forbidden_tokens` that were removed — those tokens only, with no additional entries of your own.

This is security-control reconciliation, not application feature development. Every containment threshold — how trust tiers are ranked, when a hostile flow is isolated, how repeated exposure accumulates against a subnet, and which flows cross into the responder queue — was ruled on by the flow-review board while the incident was live, and those rulings survive only in the review dossier at `/app/incident/flow_review_dossier.md`, where a later dated ruling supersedes an earlier one. Most of that log is routine incident chatter, so trace each threshold to its final ruling rather than the first mention. The exact record and digest formats the responders' tooling validates against are pinned in `/app/docs/report_spec.json`. The frozen pre-incident snapshot at `/app/workflow/.export_report.original` is read-only forensic evidence and must not be altered. Containment is judged on behaviour: it must hold against an alternate flow-telemetry capture it has never seen, reach identical decisions on repeated runs, and derive every containment call from the telemetry and the board's rulings alone.
